# M3 大量文件操作提醒设计

日期：2026-07-21
状态：设计已确认，已完成阈值、状态流转和验收边界确认，待用户复核书面稿

## 1. 背景与目标

WatchFiles 面向小米 Watch 5。手表的屏幕、系统调度和后台生命周期都不适合承诺手机级别的长时间后台文件任务，因此 M3 不继续扩大为完整的可恢复后台任务系统。

本增量只增加一个轻量的风险提示：当扫描确认任务规模较大时，在真正执行文件写入或删除前提醒用户尽量保持亮屏，并明确说明熄屏或系统调度可能导致操作中断或失败。用户可以继续，也可以取消，不改变既有文件操作安全语义。

## 2. 范围与非目标

### 本增量范围

- COPY、MOVE、DELETE 都使用扫描阶段得到的真实递归项目数和总容量判断任务规模。
- 当项目数大于或等于 100，或已知总容量大于或等于 50 MiB 时，进入统一的大任务提醒页。
- 其中 50 MiB 即 `52,428,800` bytes；项目数沿用 Scanner 的 `itemCount`，递归计入顶层项目、目录、普通文件和符号链接本身。
- `totalBytes == null` 时不得按 0 处理，只根据项目数判断是否达到阈值。
- 大任务提醒页显示扫描结果、亮屏建议、可能失败的风险和“继续操作”“取消”两个明确入口。
- COPY/MOVE 继续确认后进入现有执行流程。
- DELETE 继续确认后进入现有“永久删除”确认页，再由用户确认破坏性操作。
- 小于阈值的任务不增加额外确认步骤。
- 不改变当前操作页的卡片高度、卡片尺寸或布局；提醒使用独立圆屏状态分支并复用现有 `RoundList`/`AppChip`。
- 更新 M3 路线图、测试说明和 closeout 文档，明确记录新的手表场景边界。

### 非目标

- 不实现进程终止后的任务恢复、恢复日志、自动重试或任务持久化。
- 不声称或验收熄屏后继续执行、Activity 重入后的任务恢复或后台驻留能力。
- 不执行 5000 项大目录压力测试；大任务风险由执行前提示覆盖。
- 不改变前台 Service、Runner、Scanner、Engine、冲突处理、取消语义或路径安全规则。
- 不在本增量中调整 Android 调度策略或强制保持屏幕常亮。

冲突处理和运行中取消仍作为独立的正常文件操作验收项，不因本增量而删除。

## 3. 用户流程

### COPY/MOVE

1. 用户在目标目录页点击“放到此处”。
2. Runner 扫描源项目。
3. 扫描失败时沿用现有失败状态。
4. 扫描成功且达到阈值时显示大任务提醒；否则直接进入现有执行流程。
5. 用户选择“继续操作”后开始 Runner 的现有执行流程。
6. 用户选择“取消”时回到空闲状态，不创建终态成功或失败结果，也不执行任何文件写入。

### DELETE

1. 用户选择删除后，Runner 扫描待删除项目。
2. 扫描失败时沿用现有失败状态。
3. 扫描成功且达到阈值时先显示大任务提醒；否则直接显示现有删除预览。
4. 用户选择“继续操作”后显示现有“确认永久删除”页面。
5. 只有用户再次点击“永久删除”才进入删除执行。
6. 大任务提醒页取消时不删除任何内容，并回到空闲状态。

大任务提醒取消发生在实际任务执行前，因此不显示“已取消”终态，也不触发目录刷新或 Service 终态通知。

取消或系统返回均按同一语义处理：Runner 完成等待信号清理后回到 `Idle`，浏览页保留当前选择，不刷新目录；Service 在进入 `Idle` 后移除本任务通知并停止自身。该路径不调用 Engine，也不产生成功、失败或取消终态。

## 4. 状态与组件设计

### `FileOperationState`

新增 `WaitingForLargeOperationConfirmation` 状态，至少包含：

- 文件操作类型；
- 扫描得到的项目总数；
- 扫描得到的总容量，可为空；
- 用于继续或取消的等待信号由 Runner 内部持有，不暴露协程对象给 UI。

该状态表示“扫描已完成、文件尚未改变、等待用户确认风险”，与 DELETE 的不可逆确认和运行中的取消状态保持边界清晰。

阈值判断使用统一的纯函数 `isLargeOperation(itemCount, totalBytes)`：

- `itemCount >= 100` 时返回 `true`；
- `totalBytes != null && totalBytes >= 52,428,800` 时返回 `true`；
- 其他情况返回 `false`。

状态控制增加 `confirmLargeOperation()`。COPY/MOVE 的确认继续进入现有 Engine 执行流程；DELETE 的确认只释放大任务等待，随后仍进入现有 `WaitingForDeleteConfirmation`，不能绕过永久删除二次确认。

### `FileOperationRunner`

- 统一使用 `isLargeOperation(itemCount, totalBytes)` 判断阈值。
- `totalBytes == null` 时只根据项目数判断；不得把未知容量当成零。
- COPY/MOVE 扫描完成后，达到阈值则等待大任务确认，否则直接调用现有 `runEngine`。
- DELETE 扫描完成后，达到阈值则等待大任务确认；确认后再进入现有 `WaitingForDeleteConfirmation`。
- 新增继续和取消控制方法；继续只能释放当前大任务等待，不能启动第二个任务。
- 大任务提醒阶段的取消复用现有取消请求和等待信号清理逻辑，最终回到 `Idle`。

`FileOperationServicePort`、`FileOperationServiceGateway`、Service Client 和 Coordinator 只转发大任务确认命令，不创建第二套扫描或任务队列。Service 继续持有 Runner 和状态流；Activity 重新绑定时根据该状态重新显示大任务提醒页。

### UI

大任务提醒使用独立圆屏页面或当前操作页面的独立状态分支，不使用系统弹窗。页面至少包含：

- 标题：“文件较多”；
- 规模信息：“共 N 项 · 总计 SIZE”，容量未知时显示“大小未知”；
- 风险提示：“建议尽量保持手表亮屏。熄屏或系统调度可能导致操作中断或失败。”；
- “继续操作”；
- “取消”。

固定用户可见文案为：

- 标题：`文件较多`；
- 规模：`共 N 项 · 总计 SIZE`，容量未知时显示 `共 N 项 · 大小未知`；
- 风险：`建议尽量保持手表亮屏。熄屏或系统调度可能导致操作中断或失败。`。

系统返回键在该页等同于“取消”。继续或取消均不改变现有普通操作页、冲突页和 DELETE 永久确认页的卡片样式。

现有普通操作页、冲突页、删除确认页和终态页保持原有文案和交互。

## 5. 测试与验收

### 自动化测试

- 阈值边界：99 项且小于 50 MiB 不提醒，100 项提醒；99 项且达到 50 MiB 提醒；未知大小只按项目数判断。
- COPY/MOVE 在大任务确认前不调用 Engine，继续后只执行一次；小任务直接进入原有执行流程。
- DELETE 在大任务确认前不进入永久删除确认，继续后仍必须再次点击“永久删除”。
- 大任务提醒取消或返回回到 `Idle`，不调用 Engine、不产生终态结果、不刷新目录。
- Service、Client、Coordinator 的控制命令转发和 Activity 状态路由有 JVM 覆盖；提示标题、规模、风险文案有纯逻辑测试覆盖。
- 普通任务路径和现有冲突、取消、失败、部分成功测试继续通过。

### 文档与设备验收边界

- 保留已有 COPY/MOVE/DELETE 基础真机冒烟证据。
- 不把熄屏继续、Activity 重入任务恢复、进程恢复和 5000 项压力测试标记为待完成的产品功能；它们记录为当前手表平台边界或明确非目标。
- 真机不执行大规模压力数据生成；仅验证在受控小型 fixture 上，普通任务路径没有被新增提醒破坏。
- 真机保留已有 M3 COPY/MOVE/DELETE 和冲突/取消验收证据；不把 5000 项压力测试或熄屏后恢复能力重新列为本增量的产品验收项。
- 更新 `docs/TESTING.md`、`docs/roadmap.md`、M3 当前上下文和 closeout，避免继续把已放弃场景写成 `PENDING_DEVICE_UI`。

## 6. 完成条件

- 阈值判断只基于扫描后的真实递归统计，且未知容量处理明确。
- 大任务在文件改变前得到一次可取消的用户确认。
- DELETE 仍保留独立的永久删除二次确认。
- 普通任务、冲突处理、运行中取消和既有安全测试不回归。
- 大任务提示文案明确表达亮屏建议和可能失败风险，不承诺熄屏后台执行。
- 相关单元测试、构建和 Lint 通过。
- 项目文档与当前 M3 实际范围一致。

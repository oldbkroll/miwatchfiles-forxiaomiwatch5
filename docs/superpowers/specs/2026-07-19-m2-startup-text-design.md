# WatchFiles M2 启动基线与简易文本查看/编辑设计

日期：2026-07-19
状态：已获用户确认，待创建实施计划

## 1. 目标与边界

M2 是当前分支上的下一阶段增量，分为两个连续部分：

1. 在不改变真机文件的前提下建立启动和基础性能基线；只有基线确认瓶颈后才做针对性优化。
2. 增加简单纯文本的查看与编辑，支持覆盖当前文件和仅在当前目录内另存为。

本阶段继续保持：`targetSdk 29`、`requestLegacyExternalStorage`、`armeabi-v7a`、小米 Watch 5 自定义表冠滚动逻辑、Debug-only 开发和动态无线 ADB 发现。

本阶段不开发内置音频播放器；音频和视频保留 MIME 识别及“用其他应用打开”。视频播放、ZIP 查看和 ZIP 解压属于 M4 可选评估。文本功能不包含任意目录选择、富文本、语法高亮、编码切换、自动保存或复杂搜索。

M0、M1A、M1B、M1C 的实现、文档和验收结论不在本 spec 中重整。

## 2. 启动与基础性能专项

### 2.1 测量规则

每个设备会话开始前动态执行：

```powershell
adb devices -l
adb mdns services
```

只使用当次发现且状态为在线的 `M2505W1/grasslte` serial，不复用历史无线 ADB 地址；离线条目不使用。

基线期间不得在真机创建、删除、重命名、复制、移动、修改权限或写入任何文件。所有指标至少测量 5 次，并保留原始数据、中位数和最慢值：

- 冷启动：`force-stop` 后 `am start -W` 到首屏可交互；
- 热启动：从后台重新进入到首屏可交互；
- 主页进入普通目录到列表可见；
- 受控的大目录加载到列表可见；大目录必须使用既有内容，不为基线临时创建；
- 主页、目录页和图片查看页的 PSS；
- 连续进入/退出图片页 5 次后的 PSS 变化。

`am start -W` 的结果、页面可见时间和 `dumpsys meminfo` 输出均作为基线证据保存。基线阶段不预设绝对性能门槛，先用重复数据判断瓶颈。

### 2.2 优化范围

当前实现的 `FileBrowserViewModel.open()` 只在进入目录时触发读取，`DirectPathRepository.list()` 已在 `Dispatchers.IO` 执行；因此先验证是否确有启动或目录瓶颈。

只有发现可重复瓶颈时，才在对应路径做最小修改，优先级如下：

1. 权限页或主页先显示，避免首屏等待目录扫描；
2. 目录枚举、属性读取、排序等耗时工作继续后台执行；
3. 仅延迟初始化不需要首屏的对象；
4. 仅针对已测得的目录元数据或排序成本做优化。

每个优化点必须保留前后基线对照。不得为未验证的需求提前引入播放器、音频组件、缩略图缓存或大型依赖。

## 3. 文本识别与读取模型

### 3.1 类型范围

文本入口复用现有 `FileTypeInfo` 的 MIME 分类，并采用保守扩展名集合：`.txt`、`.text`、`.log`、`.csv`、`.md`。不对未知扩展名做内容猜测；无法确认是简单纯文本的文件继续显示普通文件详情和“用其他应用打开”。

文本读取初版只支持 UTF-8。使用严格的 UTF-8 decoder，遇到非法字节序列时不替换字符、不强行编辑，并显示“编码不支持或文件内容无效”的明确提示。

### 3.2 分段查看

文本查看使用独立的 `TextFileReader`，所有文件读取在 `Dispatchers.IO` 执行：

- 每段目标大小约 32 KiB，以字节偏移定位；
- UTF-8 多字节字符不得在页面边界被截断；
- 保留 `LF`、`CRLF` 和空行；
- 查看页面只持有当前段和必要的前后定位信息，不把整个大文件放入 Compose 状态；
- 初始可查看上限为 16 MiB，超过上限显示明确提示并建议使用其他应用；
- 读取过程中发现非法 UTF-8 时停止当前读取并显示只读/不支持提示。

分段查看页面提供有限的上一段/下一段操作，避免为了适配圆屏而引入复杂编辑器或全文搜索。

## 4. 文本页面与编辑状态

### 4.1 页面入口

在文件详情页对支持的纯文本增加“查看文本”入口。查看页在满足编辑条件时提供“编辑”操作；查看和编辑属于同一个文本功能范围，不创建独立的文本属性页。

返回路径保持现有行为：文本页返回文件详情，再返回原目录，并保留目录列表位置。

### 4.2 状态模型

文本页面至少维护以下状态：

- `Loading`：后台读取中；
- `Ready`：当前页内容、字节偏移、总大小、是否可编辑及不可编辑原因；
- `Editing`：编辑草稿、原文件快照、是否有未保存修改；
- `Saving`：保存目标、事务进度和取消可用性；
- `Saved`、`Failed` 或 `Cancelled`：显示结果并允许返回或重试。

编辑器只对 UTF-8 字节数不超过 512 KiB 的文件开放。编辑状态一次载入完整字符串，保持原始换行；超过 512 KiB 的文件保持分段只读。没有自动保存，离开有未保存修改的页面时必须明确确认。

进入编辑时记录原文件的字节摘要及基本属性。保存前重新读取并核对摘要；发现文件被其他进程修改时拒绝覆盖，要求重新打开。

## 5. 共享安全文本写入模块

覆盖和另存为都调用 `SafeTextWriteRepository`，不直接在 Compose 页面中操作 `java.nio.file.Files`。

### 5.1 目标校验与确认

写入前必须确认：

- 目标父目录就是当前浏览目录；
- 另存为名称通过 `FileNameRules`，不能包含路径分隔符、NUL、`.`、`..` 或首尾空白；
- 目标不是符号链接，父目录是可写目录；
- 覆盖当前文件或另存为命中同名目标时，界面明确显示目标名称并要求用户确认；
- 未经确认不得删除或覆盖旧目标。

另存为不提供任意目录选择，只允许当前目录内输入新文件名。取消同名确认时，源文件和已有目标都保持原内容。

### 5.2 事务写入流程

写入模块使用同目录唯一临时文件和 backup 文件，文件名包含随机事务 ID，并使用 `CREATE_NEW` 防止复用。流程如下：

1. 在应用私有事务记录中写入目标、临时文件、backup 文件、预期摘要和阶段。
2. 在目标同目录创建临时文件，分块写入 UTF-8 内容并 `force(true)`；失败或取消时只清理临时文件。
3. 如果目标不存在，将临时文件转移为目标；如果目标存在且已经得到明确覆盖确认，先将旧目标转移为 backup，再将临时文件转移为目标。
4. 发布成功后校验目标摘要，再删除 backup 和事务记录。
5. 所有 `Files.move` 均不传入 `ATOMIC_MOVE` 或 `REPLACE_EXISTING`；不使用原地截断写入。

取消只保证在发布前生效；发布完成后返回成功，不伪造“已取消”。保存失败时尽力恢复 backup，原文件内容必须保持不变；恢复失败也不得静默删除任何目标。

### 5.3 崩溃恢复与清理

事务阶段至少区分 `Staged`、`BackedUp`、`Published` 和 `Cleaned`。应用启动后的后台恢复以及下一次文本操作前的恢复逻辑遵循：

- `Staged`：原目标仍在，删除未发布的临时文件；
- `BackedUp` 且目标缺失：恢复 backup 到原目标；
- `BackedUp` 且目标摘要等于预期新内容：认定发布成功，清理 backup；
- 目标状态无法安全判断：保留原始 backup 和记录，显示明确恢复失败，不进行猜测性删除。

正常成功、取消、失败和可恢复崩溃路径都必须清理任务拥有的临时文件；不能删除用户自行创建的同名点文件或其他非本事务文件。

## 6. 测试设计

### 6.1 文本读取和页面状态

使用 JVM 临时目录测试：

- 空文件、单行、多行、`LF`、`CRLF` 和 Unicode 文本；
- 32 KiB 边界处的 UTF-8 多字节字符；
- 非法 UTF-8、不支持扩展名和超过 16 MiB 的文件；
- 512 KiB 以内可编辑、超过上限只读；
- 加载、编辑、返回、放弃、保存成功和错误状态转换。

### 6.2 保存模块 TDD 与故障注入

保存核心逻辑必须先写失败测试，再实现最小修复。通过可注入的故障点覆盖：

- 临时文件创建或写入失败；
- flush/force 失败；
- backup 转移失败；
- 新目标发布失败；
- 发布后校验失败；
- 协程取消和模拟进程中断后的恢复。

每个失败路径断言：原目标字节内容未被破坏、另存为源文件未被删除、同名目标未被静默覆盖、事务临时文件按规则清理或留下可恢复记录。

### 6.3 真机验收边界

只安装和测试 Debug APK。每次设备会话动态发现 ADB；启动基线完全只读。文本真实写入只允许使用：

`/storage/emulated/0/Download/WatchFilesTest/M1Sandbox`

真机记录源/目标内容或 SHA-256、取消和失败结果、`.part`/`.backup` 残留及 `AndroidRuntime`/`FATAL EXCEPTION` 审计。不得触碰 system、media、个人目录或 M1Sandbox 之外的路径。

阶段结束统一运行：

```powershell
:app:testDebugUnitTest
:app:assembleDebug
:app:lintDebug
git diff --check
```

## 7. 交付条件

M2 只有同时满足以下条件才可标记完成：

- 有至少 5 次重复的启动、目录加载和 PSS 基线记录；
- 只有在数据支持时才实施启动优化，并保留前后对照；
- UTF-8 文本可分段查看，小文件可编辑；
- 覆盖和当前目录内另存为共用安全写入模块；
- 同名、取消、失败、崩溃恢复和超大/不支持文本均有明确行为及测试证据；
- 真机写入严格限制在 M1Sandbox，且无残留临时文件或崩溃证据；
- 完整 Debug 单测、构建、Lint 和 diff 检查通过；
- 更新 M2 专属 checkpoint、必要测试清单和路线图状态后，提交一个可解释的 M2 阶段增量。

# WatchFiles M1C 删除确认设计

日期：2026-07-17  
状态：设计已由用户确认，spec 待用户审阅

## 1. 目标

M1C 为 WatchFiles 增加面向圆屏的递归永久删除能力，并在任何破坏性写操作之前提供清晰、独立的二次确认。

本阶段必须满足：

- 选择一个或多个文件、文件夹后发起删除。
- 删除前扫描并显示项目数和可统计的总大小。
- 删除确认页明确提示“永久删除，无法恢复”。
- 删除任务递归、深度优先执行，不跟随符号链接。
- 内部存储根目录不可删除。
- 删除中可协作式取消；已经删除的内容不回滚，尚未处理的内容保留。
- 失败和部分成功必须在圆屏上可理解，并保留失败详情。

## 2. 范围边界

本阶段只保证应用进程存活期间的前台删除任务。回收站、撤销删除、息屏继续、前台服务、进程被杀后的恢复和恢复日志不在 M1C 范围，分别留给后续阶段。

删除仍使用现有单任务约束：同一时间最多运行一个 COPY、MOVE 或 DELETE 任务。删除确认扫描本身也占用当前任务槽，不能在确认页期间发起第二个文件任务。

真机写操作仍只在 `/storage/emulated/0/Download/WatchFilesTest/M1Sandbox` 内验收。该目录是测试限制，不把 M1Sandbox 硬编码为应用的生产删除范围。

本阶段保持以下现有约束不变：

- `targetSdk 29`
- `android:requestLegacyExternalStorage="true"`
- 小米 Watch 5 自定义表冠滚动兼容逻辑：`rotaryScrollableBehavior = null` 与 `onRotaryScrollEvent`
- 仅打包 `armeabi-v7a`
- 开发期间只构建 Debug；M1C 完成且 M1 全部验收后才考虑 Release
- 每次设备测试前动态发现无线 ADB，不假定历史地址或序列号

## 3. 方案决策

### 3.1 复用现有文件操作架构

在现有 `FileOperationEngine`、`FileOperationCoordinator`、`FileOperationState` 和圆屏操作页上增加 DELETE 分支，不建立第二套删除队列。这样 COPY、MOVE、DELETE 共用单任务互斥、进度、取消和最终结果映射。

为避免重构已经通过 M1B 验收的 COPY/MOVE 调用链，保留现有 `FileOperationRequest` 的字段顺序，并将 `targetDirectory` 改为仅对 DELETE 可为空的字段；同时提供明确的 transfer/delete 构造函数。COPY/MOVE 构造和扫描阶段必须拒绝空目标目录，DELETE 构造必须传入空目标目录，删除分支绝不解析或使用目标目录，避免把删除误当成目标覆盖操作。

### 3.2 独立确认状态

协调器增加删除准备流程：

1. `prepareDelete(sources)` 生成任务 ID，进入扫描状态。
2. 扫描成功后进入 `WaitingForDeleteConfirmation`，只暴露预览摘要，不调用删除引擎。
3. 用户点击“永久删除”后才进入 `Running(DELETE)` 并执行删除。
4. 用户在确认页点击“取消”或按系统返回键时，任务回到空闲状态并返回浏览页，不产生任何文件变化。

扫描失败直接显示 DELETE 失败结果，不出现可执行的永久删除按钮。

## 4. 用户流程与圆屏界面

### 4.1 从多选进入确认

选择模式增加“删除”入口，副标题说明这是永久删除。点击后保留当前选择并启动删除预扫描。

确认页使用现有 `RoundList` 和自定义表冠滚动，不使用容易被圆形边缘裁切的小型系统弹窗。页面至少包含：

- 标题：`确认永久删除`
- 项目数量：选中的顶层项目数，以及扫描得到的总项目数
- 大小：可统计时显示总大小；有项目大小无法读取时显示“大小未知”
- 警告：`永久删除，无法恢复`
- 主操作：`永久删除`
- 次操作：`取消`

确认页显示的是扫描结果，不把预扫描当作删除成功。主操作在用户点击前不产生任何删除副作用。

### 4.2 删除进行中

删除操作页复用现有文件操作页，并增加 DELETE 文案：

- 标题：`正在删除`
- 当前项目、已处理项目数/总项目数
- 已处理字节数；总大小未知时不伪造精确百分比
- 取消按钮，说明“停止后已删除内容无法恢复”

删除运行期间系统返回键不退出操作页；取消通过协作式 token 传递给引擎，等待当前删除调用结束后进入结果状态。

### 4.3 结果页

复用现有结果页显示完成数、失败数和首条简短失败原因。DELETE 的结果必须区分：

- 全部成功：所有顶层项目均已删除。
- 已取消：部分项目可能已经删除，未开始的项目保留。
- 部分成功：部分顶层项目删除完成，其他项目失败；某个目录可能已经删除部分子项但自身仍保留。
- 失败：没有项目成功删除，或扫描/启动阶段即失败。

结果页提供“返回目录”，返回后刷新当前目录并清理无效选择。

## 5. 删除安全规则

### 5.1 预扫描与边界保护

扫描器在 `Dispatchers.IO` 执行，使用规范化绝对路径和 `NOFOLLOW_LINKS` 检查：

- 拒绝空选择、已不存在的源项目和不可读源项目。
- 拒绝内部存储根目录本身；引擎再次执行同一保护，形成防御性校验。
- 递归统计每个顶层源、目录、普通文件和符号链接本身。
- 可读取的普通文件累加大小；任一大小读取失败时保留项目计数并将总大小标记为未知。
- 不使用 `toRealPath()` 追踪可能存在的符号链接，也不把源路径解析到链接目标。

扫描只读取文件系统，不删除、不重命名、不清理任何临时文件。

### 5.2 递归删除

删除引擎对每个顶层源执行无符号链接跟随的深度优先删除：

1. 删除目录中的子文件和子链接。
2. 删除空的子目录。
3. 最后删除顶层目录。

符号链接按单个目录项处理，只删除链接本身，不访问链接目标。删除逻辑只接收用户明确选中的顶层路径和其递归子项，不按 `.part`、`.backup` 或其他后缀扫描清理，因此不会触碰用户自己的同名文件。

删除每个目录项前以及每个顶层源开始前检查取消 token。当前目录项删除调用完成后才响应取消；已删除内容不恢复，也不为了伪造“未发生”而删除其他目标。

### 5.3 失败和部分完成

删除单个文件失败时记录该源并继续处理后续顶层源，除非收到取消。递归目录中某个子项失败时记录失败、继续删除其他可安全处理的兄弟项，只有确认顶层路径已经不存在时才将该顶层项目计为完成。

如果目录只删除了部分内容，结果必须明确提示“目录删除未完成，部分内容可能已删除”，不能报告为完整成功。源项目在扫描后消失时报告“源项目已不存在”；权限异常报告“没有权限删除”；其他异常报告“删除失败”，同时保留技术原因供日志和测试使用。

结果统计沿用现有模型：`OperationProgress` 的项目数覆盖扫描到的全部目录项，最终结果的完成数和失败数按顶层选中项目统计。部分删除的顶层目录计为失败并附带明确说明。

## 6. 组件与数据流

### `FileOperationModels`

- 增加 `FileOperationType.DELETE`。
- 保持 `FileOperationRequest(taskId, type, sources, targetDirectory)` 的现有参数顺序；仅将目标目录设为 DELETE 可为空，并增加 `transfer(...)` 与 `delete(...)` 构造函数。
- 增加 `DeletePreview(topLevelCount, itemCount, totalBytes)`，其中 `totalBytes` 可为空。
- 增加 `WaitingForDeleteConfirmation(preview)` 状态；该状态不能被操作页误认为已开始删除。
- 保留现有 COPY/MOVE 状态和结果类型的语义。

### `FileOperationScanner`

保留 COPY/MOVE 的目标目录校验逻辑；DELETE 分支跳过目标校验，执行根目录保护、源有效性检查和递归统计。扫描取消不会调用删除引擎。

### `FileOperationEngine`

增加 DELETE 执行分支和可注入的 `storageRoot`、删除原语，使以下行为可在 JVM 测试中验证：

- 深度优先顺序
- 不跟随符号链接
- 删除失败后的继续处理和结果分类
- 取消后停止后续删除
- 根目录保护的二次校验

COPY/MOVE 的 staging、替换事务和快速移动路径保持原样。

### `FileOperationCoordinator`

增加 `prepareDelete(sources)`、`confirmDelete()` 和删除确认状态管理。确认页期间拒绝第二个任务；未确认取消只清理协调器状态，不产生终态文件操作结果。确认后沿用现有扫描、引擎进度、取消、部分成功和终态映射。

### `MainActivity` 与 `FileOperationScreens`

增加删除确认路由和删除入口，复用现有圆屏安全区域、表冠滚动和结果页。确认页取消/返回只回浏览页；执行中返回键保持禁用；终态返回目录时刷新浏览列表并消费结果。

## 7. TDD 与自动化验证

在生产代码前先添加失败测试，至少覆盖：

### 预扫描

- 普通文件、空文件、空目录和三层以上嵌套目录的项目数/大小统计。
- 根目录拒绝。
- COPY/MOVE 缺少目标目录以及 DELETE 携带目标目录时均被拒绝。
- 源项目消失、不可读和大小未知处理。
- 符号链接只计为单个项目，不遍历链接目标。

### 删除引擎

- 普通文件删除后不存在。
- 递归目录完整删除，空目录和混合目录均正确处理。
- 符号链接只删除链接本身，链接目标保持不变。
- 子项删除失败时不伪造顶层成功，并返回失败详情。
- 删除过程中取消，已处理内容保持删除，未处理内容保持存在。
- 删除逻辑不触碰用户创建的 `notes.part` 等无关文件。

### 协调器与确认

- 未调用 `confirmDelete` 前绝不调用删除引擎。
- 确认页取消/返回不会删除任何内容。
- 确认期间第二个任务启动被拒绝。
- 确认后正确映射成功、取消、失败和部分成功状态。
- 终态消费后返回 `Idle`。

本地验证命令：

```powershell
.\gradlew.bat :app:testDebugUnitTest --no-daemon --console=plain
.\gradlew.bat :app:assembleDebug :app:lintDebug --no-daemon --console=plain
git diff --check
```

开发期不构建 Release，不更新版本号或 `releases` APK。

## 8. 小米 Watch 5 真机验收

所有写入严格限制在 `/storage/emulated/0/Download/WatchFilesTest/M1Sandbox`。每次测试前执行：

```powershell
adb devices -l
adb mdns services
```

只选择当前动态发现且型号为 `M2505W1`/设备为 `grasslte` 的在线无线 ADB 传输，不复用历史地址。

验收至少覆盖：

1. 在 M1Sandbox 中选择普通文件、空目录和嵌套目录，点击删除后确认页显示正确数量和大小。
2. 点击“取消”或系统返回，确认所有源项目仍存在且没有删除引擎运行。
3. 点击“永久删除”，确认普通文件、空目录和递归目录均被删除，当前列表刷新正确。
4. 在大目录删除过程中点击取消，确认已删除部分不恢复，未开始项目仍存在，结果页明确说明不可恢复。
5. 通过本地注入失败测试验证权限/子项失败的部分成功语义；不通过修改真实重要目录权限制造风险。
6. 检查 `M1Sandbox` 中没有因 DELETE 产生的临时残留，用户自己的 `notes.part` 保持不变。
7. 检查 `AndroidRuntime`、`FATAL EXCEPTION` 为空，并回归验证 M1A/M1B、表冠滚动和图片查看器。

## 9. 完成条件与非目标

M1C 完成条件：自动化测试、Debug 构建和 Lint 通过；真机删除确认、递归删除、取消、列表刷新和无崩溃检查通过；根目录保护和失败/部分成功结果有测试证据；Git 工作区干净并提交阶段文档。

本阶段不实现回收站、撤销删除、后台服务、进程恢复、音视频播放器、Release 包或 M2/M3 功能。

# WatchFiles M2-M4 范围与简化工作流设计

日期：2026-07-18
状态：已完成方案讨论，待用户审阅本设计文档

## 1. 目标

在保持 Watch 5 真机安全底线的前提下，收敛 M2-M4 的产品范围，并把开发流程从固定强度调整为按风险分级。目标是优先完成手表上真正有价值、可维护、可验证的能力，同时减少低价值功能和重复流程带来的开发成本。

本设计不修改 M0、M1A、M1B、M1C 的验收结论，也不改变既有设备、构建和存储安全约束。

## 2. 已确认的产品范围

### M2：启动优化与简易文本查看/编辑

M2 的第一个小增量是启动速度和基础性能基线，之后完成简易文本查看与编辑。

文本功能范围：

- 支持 `.txt`、`text/plain` 等简单纯文本；
- 查看和编辑放在同一功能范围；
- 支持覆盖当前文件；
- 支持仅在当前目录内输入新文件名后另存为；
- 覆盖和另存为共用同一套安全文本写入模块；
- 不支持任意目录选择、富文本、语法高亮、编码切换、自动保存和复杂搜索；
- 超过可安全编辑范围的文本不得强行载入编辑器，应保持只读或明确提示不支持；
- 初版优先支持 UTF-8，其他编码不作为首期目标。

文本查看属于 B 级；覆盖和另存为属于 C 级，因为会修改真实文件。

音频不再开发内置播放器。现有 MIME 识别和“用其他应用打开”能力继续保留，音频文件通过手表已有应用播放。

### M3：可靠性、后台操作与性能

M3 保留与核心文件管理体验直接相关的能力：

- 前台文件操作服务；
- 息屏继续文件操作；
- 操作恢复日志；
- 启动速度、内存和目录加载优化；
- 5000 项大目录压力测试；
- 表冠交互和厂商触觉反馈兼容性收尾。

以下功能从路线中删除，不再作为早期版本目标：

- 收藏目录；
- 最近目录；
- 文件系统搜索；
- 用户可选排序；
- 独立文件属性页。

固定的“文件夹优先、名称排序”继续保留，它是基础目录呈现规则，不属于新增的用户排序功能。

文件详情页继续显示已有的 MIME、大小、修改时间、读写状态和路径。MIME 只用于表达文件类型，不替代完整文件元数据，也不新增独立属性页面。

### M4：可选的高级本地能力

M4 只保留为后续评估阶段，不设为核心版本必须完成的交付：

- 内置视频播放；
- ZIP 查看；
- ZIP 解压。

视频在 M2-M3 期间继续通过“用其他应用打开”。M4 的内置视频播放只在核心文件操作和性能稳定后，根据真机编解码、内存、电量和圆屏交互结果决定是否实现。

ZIP 查看和 ZIP 解压需要分开评估：读取压缩包目录属于只读能力；解压会产生真实文件写入，必须单独处理路径穿越、同名覆盖、压缩炸弹、空间不足、取消、临时目录和部分成功结果。ZIP 解压按 C 级功能处理，不因属于 ZIP 功能而降低安全要求。

以下能力明确不进入早期版本：

- 内置音频播放器；
- APK 信息及安装；
- 局域网/WebDAV；
- 可选 Shizuku/root 后端。

## 3. 启动与基础性能专项

启动优化作为 M2 的第一个小增量，先测量再修改，不预先设定未经真机验证的绝对阈值。

### 基线指标

动态发现当次在线无线 ADB serial 后，分别重复冷启动和热启动测试，记录：

- 冷启动到首屏可交互时间；
- 热启动到首屏可交互时间；
- 从主页进入目录到列表可见时间；
- 大目录加载时间；
- 主页、目录页和图片查看页的 PSS；
- 连续进入/退出页面后的内存变化。

每项至少记录 5 次，保留中位数和最慢值。只有在基线确认瓶颈后才实施针对性优化。

### 优化方向

- 启动阶段优先显示权限页或轻量主页，不提前扫描存储目录；
- 目录读取、文件属性读取、图片解码等耗时工作保持在后台；
- 延迟初始化不需要首屏的 ViewModel、扫描器、媒体资源和缓存；
- 大目录优化优先关注条目元数据读取和排序成本；
- 不为尚未验证的需求提前引入播放器、缩略图缓存或大型媒体组件；
- 每个优化点都要能通过前后基线数据说明收益。

## 4. 风险分级工作流

### A 级：低风险改动

适用于文档、文案、颜色、间距和不改变行为的重命名。

- 读取当前任务相关文档；
- 用几行说明目标和验收条件；
- 不强制先写测试；
- 按需做 Markdown、静态检查或手工检查；
- 文档-only 改动不构建 APK；
- 不要求独立复审和真机写操作。

### B 级：中风险改动

适用于启动优化、文本查看、页面导航、列表交互、表冠交互、MIME 判断、图片查看和不涉及真实写入的媒体能力。

- 使用 3-5 行小计划，不强制拆成完整 spec、plan 和逐任务 brief；
- 关键状态转换和容易回归的逻辑先补单测；普通 UI 按需要补测试；
- 开发中运行聚焦测试；
- 涉及手表交互时做一次动态 ADB 真机只读检查；
- 阶段收尾统一运行完整 Debug gate；
- 只有跨模块或阶段性功能才安排独立复审。

### C 级：高风险改动

适用于文本覆盖/另存为、DELETE、COPY、MOVE、权限、路径安全、取消、失败恢复、临时文件、前台文件操作服务、真实设备写入和 ZIP 解压。

- 保留设计确认、TDD、故障注入测试和必要的独立复审；
- 真机写操作严格限制在 `/storage/emulated/0/Download/WatchFilesTest/M1Sandbox`；
- 每次设备会话动态发现无线 ADB，不复用历史地址；
- 记录界面结果、源/目标哈希、残留文件和 crash audit；
- 阶段结束前运行完整 Debug 单测、`assembleDebug`、`lintDebug` 和 `git diff --check`；
- 必要时保留独立 checkpoint，不以提速为理由省略安全证据。

### 跨阶段规则

- 每阶段使用一个独立会话；
- 开发期间只构建和安装 Debug；
- 每个提交对应一个完整功能、可解释修复或阶段收尾，不要求每个微任务单独提交；
- 已关闭阶段的逐任务 brief、review diff 和临时日志不作为新会话默认上下文；
- 不重新初始化项目，不撤销 `targetSdk 29`、`requestLegacyExternalStorage`、`armeabi-v7a` 或小米手表自定义表冠滚动兼容逻辑。

## 5. 不变的安全底线

- DELETE、COPY、MOVE、文本保存、ZIP 解压和所有真实文件写入继续使用 TDD；
- 删除必须经过确认，保留 root guard、无链接跟随、不可逆取消和明确失败分类；
- 真机写操作仅限 M1Sandbox；
- 每次动态发现无线 ADB 在线 serial；
- 生产移动/发布路径不得使用 `ATOMIC_MOVE` 或 `REPLACE_EXISTING`，不得先删除未经确认的旧目标；
- 阶段完成后才进行 Release 交接。
